## 1. 현재 방식의 구조적 한계

### 1.1 측정 파이프라인 진단

현재 방식은 **거리 추정 → 포인트 캡처**를 시간적으로 분리합니다.

```
[QR 인식 단계]         [이동]        [측정 단계]
QR → d 잠금   ─────────────────→   터치 2회 (height)
                                  → 화면 전환
                                  → 터치 2회 (width)
```

이 구조에서 발생하는 오차 원천 5가지:

**① FOV 고정 가정 (`detector.js:25`)**

```js
const FOCAL_LENGTH = 554;  // 60° FOV 가정, 기기마다 다름
```

실제 모바일 카메라 FOV는 55°~80° 범위로 분포. FOCAL_LENGTH 오차 → 거리 오차 → 높이 오차로 직결.

| 실제 FOV | 실제 f | 코드의 f | 거리 오차 | 5m 나무에서 |
|----------|--------|----------|-----------|-------------|
| 55° | 614 | 554 | −10% | −50cm |
| 60° | 554 | 554 | 0% | ±0 |
| 70° | 457 | 554 | +21% | +105cm |
| 80° | 381 | 554 | +45% | +225cm |

**② QR 상단 1변만 사용 (`detector.js:129-131`)**

```js
const dx = loc.topRightCorner.x - loc.topLeftCorner.x;
const dy = loc.topRightCorner.y - loc.topLeftCorner.y;
const pixelSizeDetect = Math.sqrt(dx*dx + dy*dy);
```

4개 코너 중 상단 변 하나만 픽셀 길이 측정. QR이 사선으로 촬영되면 투영 길이가 짧아져 거리 과대 추정.

**③ 2단계 측정의 누적 오차**

높이(2터치) → 폭(2터치) = 총 4터치, 2화면. 각 터치마다 ±5~15px 오차 발생.

**④ 잠금 거리(lockedDistance)의 구조적 문제**

QR 인식 후 사용자가 뒤로 물러나면 거리가 바뀌지만 lockedDistance는 업데이트되지 않음.

**⑤ 카메라 기울기 미보정**

전체 나무를 카메라에 담으려면 폰을 위로 들어야 하는데, 현재 공식은 카메라가 완전히 수평일 때만 정확. 상세는 [5장](#5-기울기-보정-설계) 참조.

## 2. 새로운 측정 패러다임

### 2.1 핵심 아이디어

> **QR 코드 4개 코너 = 카메라 보정 데이터**
>
> 20×20cm QR의 4개 코너 좌표를 알고, 이미지에서 4개 코너 픽셀 위치를 알면
> → **카메라 내부 파라미터(f) + 외부 파라미터(R, t)를 동시에 계산** 가능.

이것을 **호모그래피 분해(Homography Decomposition)** 라 합니다.

현재 방식이 "거리만" 구하는 것과 달리, 새 방식은 **카메라-QR 간의 완전한 3D 좌표계**를 수립합니다.

### 2.2 새로운 측정 흐름

```
[Phase 1] QR 스캔
  QR 4코너 검출 → 호모그래피 계산 → f 보정 + 3D 좌표계 수립
                                         ↓
[Phase 2] 후퇴 & 구도 잡기
  사용자가 뒤로 물러남 → QR 추적하며 실시간 d, f 업데이트
  + DeviceOrientation 실시간 수신
  → 전체 나무가 보이면 [📷 프레임 고정] 버튼
  → d, R_capture 동시 스냅샷
                                         ↓
[Phase 3] 3점 단일 촬영
  터치 1: 꼭대기 (p₁)
  터치 2: 밑동   (p₂)   ┐
  터치 3: 수관 끝 (p₃)  ┘ → 광선 교점 계산 → 높이 + 폭 동시
                                         ↓
  result.html (높이 + 폭)
```

**핵심 개선점**:
- 측정을 **단일 프레임**에서 완료
- QR에서 얻은 **보정된 f값** 사용
- 프레임 고정 시점의 **기울기(R_capture)** 로 3D 광선 보정

### 2.3 Phase 2 QR 소실 처리

사용자가 뒤로 물러나는 도중 QR이 나뭇가지·풀숲에 가려지거나 화면 밖으로 나갈 수 있습니다.

```
QR 가시 상태에 따른 처리 방침:

QR 보임(pixelSize ≥ 80px) → 매 프레임 d·f 업데이트
QR 보임(pixelSize < 80px) → 마지막 안정 d·f 유지 + "QR 작음" 경고
QR 소실(LOST)             → 마지막 안정 d·f 잠금 + 주황 경고 배지 표시
                           → [📷 프레임 고정] 버튼 활성화 유지
                             (이미 QR에서 d·f를 얻었으므로 측정 진행 가능)
```

**QR 소실 후 측정 정확도**: 소실 직전 마지막으로 안정적으로 계산된 d·f를 사용하므로 측정은 가능하지만, 소실 시점과 프레임 고정 시점 사이에 거리가 변하면 오차 발생. UI에서 "거리 미갱신" 경고로 사용자 인지 유도.

|------|------|------|------|--------------|

| **WebXR API** | 기기 AR 포즈 | OS 수준 정밀도 | iOS Safari 미지원 | ⚠️ Android only |
| **MindAR.js** | 이미지 추적 AR | 쉬운 API | QR 특화 아님 | ✅ |
| **jsQR + 수작업 DLT** | 호모그래피 직접 구현 | 의존성 없음, 경량 | 구현 복잡 | ✅ |

### 추천 조합

```
jsQR              → QR 데이터 + 4코너 픽셀 좌표 추출     (현재 유지)
Three.js          → 3D 좌표계·오버레이 렌더링
DLT 직접 구현     → 호모그래피 계산 (OpenCV.js 없이 경량화)
DeviceOrientation → 기울기 보정용 회전 행렬              (신규 통합)
```

**OpenCV.js를 쓰지 않는 이유**: 4점 호모그래피(DLT 8방정식)는 순수 JS로 300줄 이내 구현 가능. 7MB 라이브러리 의존 불필요.

## 4. 수학적 모델

### 4.1 Phase 1: QR 호모그래피로 카메라 보정

**입력 A — QR 4코너의 3D 실세계 좌표** (단위: m, QR 중심 = 원점)

```
Q₁ = (−0.10, +0.10, 0)   좌상
Q₂ = (+0.10, +0.10, 0)   우상
Q₃ = (+0.10, −0.10, 0)   우하
Q₄ = (−0.10, −0.10, 0)   좌하
```

**입력 B** — jsQR이 제공하는 4코너 픽셀 좌표 (`loc.topLeftCorner` 등)

**DLT(Direct Linear Transform) 알고리즘**으로 3×3 호모그래피 행렬 H를 계산:

```
[u]     [h₁₁  h₁₂  h₁₃] [X]
[v]  =  [h₂₁  h₂₂  h₂₃] [Y]
[1]     [h₃₁  h₃₂  h₃₃] [1]
```

**H에서 초점거리 f 추출 — 전제 조건**

H = K × [r₁ r₂ t] (K는 카메라 행렬)에서 f를 추출하려면 두 가지를 가정합니다:

1. **cx = W/2, cy = H/2** — 주점(principal point)이 이미지 중심에 있음. 모바일 카메라의 경우 이 가정이 1~2% 오차 내에서 성립합니다.
2. **픽셀 종횡비 = 1** (정사각형 픽셀) — 사실상 모든 현대 카메라에서 성립.

두 가정 하에 직교 제약 조건 두 개에서 f를 유일하게 계산할 수 있습니다:
```
h₁ᵀ (K⁻ᵀ K⁻¹) h₂ = 0          (r₁ · r₂ = 0)
h₁ᵀ (K⁻ᵀ K⁻¹) h₁ = h₂ᵀ (K⁻ᵀ K⁻¹) h₂   (|r₁| = |r₂|)
```

**f 값은 매 프레임 계산하여 이동 평균으로 안정화**합니다. QR이 가깝고 크게 보일수록 계산 신뢰도가 높아지므로 Phase 2 초반(QR이 큰 상태)에서 충분한 프레임을 수집합니다.

H에서 분해 결과:
- **초점거리 f** — 기기 실제값, FOV 가정 없음
- **카메라-QR 회전·이동 (R₀, t₀)** — 3D 포즈, `d₀ = |t₀|`

### 4.2 Phase 2: 후퇴 중 포즈 업데이트

QR이 계속 보이는 동안 매 프레임 호모그래피 재계산 → 실시간 d 업데이트.

```
d = |t|          (이동 벡터의 크기)
f_avg = EMA(f)   (지수 이동 평균, 안정화)
```

QR 픽셀 크기(상단 변 길이)가 **80px 미만**으로 작아지면 계산 신뢰도가 급락하므로 해당 프레임의 f, d 업데이트를 중단하고 마지막 안정값을 유지합니다.

[📷 프레임 고정] 버튼 클릭 시 **d_lock, f_lock, R_capture 동시 스냅샷** (기울기 보정 핵심).

### 4.3 Phase 3: 3점 측정 (기울기 보정 포함)

상세 보정 방법은 [5장](#5-기울기-보정-설계) 참조. 기울기 보정이 적용된 최종 측정 공식:

```
// 각 터치 픽셀 (uᵢ, vᵢ) → 3D 세계 좌표 Pᵢ

ray_cam_i = normalize( (uᵢ − cx)/f_lock,  (vᵢ − cy)/f_lock,  1 )
ray_world_i = R_capture × R_cam_to_device × ray_cam_i
t_i         = d_lock / ray_world_i.x       // 나무 수직면(x = d)과 교점 파라미터
P_i_world   = ray_world_i × t_i

// 측정
높이 = |P₁_world.z − P₂_world.z|
폭   = |P₃_world.y − P_trunk_y| × 2
```

**`P_trunk_y` 정의**: 꼭대기(P₁)와 밑동(P₂)의 수평 Y좌표 평균값.

```
P_trunk_y = (P₁_world.y + P₂_world.y) / 2
```

나무가 수직으로 곧게 서 있으면 P₁.y ≈ P₂.y이므로 차이가 미미하지만, 기울어진 나무에서는 두 값의 평균을 trunk 중심축으로 사용합니다.

**나무 수직면 정의**: 현재 공식은 `x = d_lock` (카메라 정면 방향으로 d 거리의 수직면)을 가정합니다. 이 가정은 사용자가 QR 정면에서 뒤로 물러날 때 성립합니다. 대각선 방향으로 물러나는 경우 plane 법선 벡터를 Phase 1의 t₀ 방향으로 정의하는 방식으로 개선할 수 있습니다.

## 5. 기울기 보정 설계

### 5.1 문제: 기울기가 없으면 측정이 틀린 이유

`h = d × Δy / f` 공식은 카메라가 **수평 방향을 바라볼 때만** 정확합니다.

전체 나무(10m)를 5m 거리에서 보려면 카메라를 **약 20~30° 위로** 들어야 합니다. 이 상태에서 naive 공식을 사용하면 꼭대기를 보는 광선의 실제 각도가 무시되어 높이가 **과소 추정**됩니다.

**기울기별 오차 시뮬레이션**

아래 수치는 다음 조건에서 수치적으로 계산한 값입니다:
- 카메라 위치: 지면 (h_c = 0m)
- 나무 높이: H = 10m, 거리 d = 5m
- 공식: h_naive = d × [y_top − y_base] / f (각 픽셀 위치는 카메라 회전 후 투영)

| 상향 기울기 θ | Naive 측정값 | 실제 높이 | 오차 |
|--------------|-------------|-----------|------|
| 0° (수평) | 10.0m | 10.0m | **0%** |
| 15° | 7.0m | 10.0m | **−30%** |
| 25° | 6.3m | 10.0m | **−37%** |
| 35° | 6.2m | 10.0m | **−38%** |

> **주의**: 위 수치는 카메라가 지면 높이에 있을 때의 계산값입니다. 카메라 높이(통상 눈높이 ~1.5m)와 나무 높이·거리 비율에 따라 오차 크기가 달라집니다. 공통적인 결론은 **기울기가 커질수록 나이브 공식은 심각하게 과소 추정**한다는 것이며, 오차는 기하 조건에 따라 10~40% 범위에 분포합니다.

> 전체 나무가 보일 정도로 뒤로 물러나도 폰을 수평으로 유지하기 어렵기 때문에
> **이 오차는 측정 시 거의 항상 발생합니다.**

### 5.2 DeviceOrientation 데이터 구조 및 좌표계

```
DeviceOrientation 이벤트:
  alpha (α): 나침반 방향 (yaw,  0~360°)
  beta  (β): 앞뒤 기울기  (pitch, −180~180°, 수직=90°)
  gamma (γ): 좌우 기울기  (roll,  −90~90°)
```

W3C 사양 기준 **기기 프레임 → 세계 프레임** 변환 행렬:

```
R_device_to_world = Rz(−α) × Rx(−β) × Ry(γ)
```

**좌표계 불일치 주의**

DeviceOrientation의 R_device_to_world는 **기기(device) 프레임을 세계 프레임으로** 변환합니다. 그런데 카메라 좌표계는 기기 좌표계와 다릅니다:

```
기기 좌표계 (portrait 모드):
  X_device → 오른쪽
  Y_device ↑ 위 (스크린 상단)
  Z_device → 사용자 방향 (스크린 밖으로)

카메라 좌표계 (표준 핀홀):
  X_cam → 오른쪽      (= +X_device)
  Y_cam ↓ 아래        (= −Y_device)
  Z_cam → 장면 방향   (= −Z_device)
```

따라서 카메라 광선을 세계 좌표로 변환할 때 중간 변환이 필요합니다:

```
R_cam_to_device = [1,  0,  0]
                  [0, −1,  0]
                  [0,  0, −1]

ray_world = R_device_to_world × R_cam_to_device × ray_cam
```

> **검증 필수**: 위 R_cam_to_device는 일반적인 후면 카메라 portrait 모드의 통상적 값입니다. 기기 제조사마다 차이가 있을 수 있으므로 **알려진 방향으로 캘리브레이션 테스트**를 통해 부호를 확인해야 합니다.

### 5.3 보정 수준 3단계 비교

#### Level 1 — 단순 피치 보정 (근사, 구현 쉬움)

```js
const pitchRad = (beta - 90) * Math.PI / 180;
h_corrected = h_naive / Math.cos(pitchRad);
```

- 구현 복잡도: 낮음 (5줄)
- 정확도 개선: 대략적
- 한계: 이미지 위치별 보정 차이 무시, 롤(gamma) 미반영, cos 단일 인수는 근사값

#### Level 2 — 광선 교점 계산 ★ 추천

프레임 고정 시 스냅샷한 `R_capture`로 각 터치 픽셀을 3D 광선으로 변환, 나무 수직면과의 교점으로 실세계 좌표를 계산합니다.

```
[터치 픽셀 (u, v)]
        ↓
정규화 카메라 좌표:
  x_c = (u − cx) / f_lock
  y_c = (v − cy) / f_lock
  ray_cam = normalize(x_c, y_c, 1)
        ↓
기기 좌표계로 변환:
  ray_device = R_cam_to_device × ray_cam
        ↓
세계 좌표로 변환:
  ray_world = R_device_to_world × ray_device
        ↓
나무 수직면(x = d_lock)과 교점:
  t         = d_lock / ray_world.x
  P_world   = ray_world × t
```

세 점의 P_world에서:

```
높이 = |P₁_world.z − P₂_world.z|
폭   = |P₃_world.y − P_trunk_y| × 2
```

- 구현 복잡도: 중간 (~80줄)
- 기울기 오차 완전 제거
- 롤(gamma) 자동 처리
- [📷 프레임 고정] 순간만 스냅샷 → **측정 중 폰 흔들림 영향 없음**

#### Level 3 — QR 포즈 + 센서 융합 (최고 정밀도)

Phase 1에서 구한 R₀(호모그래피 기반)와 Phase 3 시점의 DeviceOrientation을 융합합니다:

```
Phase 1 QR 인식:   R₀, t₀  (호모그래피)
프레임 고정 시:    R_capture = R_delta × R₀

여기서:
  R_delta      = R_sensor_phase3 × R_sensor_phase1⁻¹
  R_sensor_t   = R_device_to_world(α_t, β_t, γ_t)  at time t
```

- 구현 복잡도: 높음 (쿼터니언 적분 또는 연속 호모그래피 추적 필요)
- 조건: QR이 계속 보이거나 센서 fusion 드리프트가 작아야 함

### 5.4 추천 및 구현 통합 위치

**Level 2 채택** — `homography.js`에 두 함수 추가:

```js
// 회전 행렬 생성 (DeviceOrientation → 3×3 matrix)
// R_device_to_world 반환
Homography.buildRotationMatrix(alpha, beta, gamma)  // → Float64Array[9]

// 터치 픽셀 → 세계 좌표 교점 (cam↔device 변환 내장)
Homography.intersectTreePlane(u, v, f, cx, cy, R_device_world, d)  // → {x, y, z}
```

`app-3p.js`의 [📷 프레임 고정] 핸들러:

```js
// 기존 (거리만 잠금)
lockedDistance = Detector.getDistance();

// 변경 (거리 + f + 회전 행렬 동시 잠금)
lockedDistance = calibrator.getDistance();
lockedFocal    = calibrator.getFocal();      // 다중 프레임 평균 f

    currentAlpha, currentBeta, currentGamma
);
```

### 5.5 롤(gamma) 보정의 추가 이점

롤 보정 없이 폰이 기울어진 상태에서 측정하면 이미지 축과 세계 축이 불일치합니다:

```
롤 0°:   이미지 Y축 = 세계 상하축  (정확)
롤 10°:  cos(10°)=0.985, sin(10°)=0.174
         → 이미지 Y축 = 세계 상하 98.5% + 수평 17.4% (오염)
롤 20°:  → 이미지 Y축 = 세계 상하 94% + 수평 34%
         → 높이 측정값에 수평 성분 혼입, 약 6% 오차 추가
```

Level 2 광선 교점 방식은 롤 오차를 **자동으로 완전 제거**합니다.

### 7.4 Phase 2 기울기 안정성 인디케이터

```
기울기 안정성 계산:
  최근 1초(약 50샘플)의 beta, gamma 분산을 계산
  variance = Var(beta_samples) + Var(gamma_samples)

  variance < 0.5°² → "안정 (초록)"   → [프레임 고정] 버튼 강조
  0.5 ~ 2.0°²     → "주의 (노랑)"
  > 2.0°²         → "불안정 (빨강)"  → 버튼 반투명 + 경고 텍스트
```

기울기 안정성 표시 이유: 흔들리는 상태에서 R_capture를 스냅샷하면 오차가 커지므로 사용자에게 안정 후 고정을 유도합니다.

### 9.3 남는 구조적 한계

```
[물리적 한계]
  - 수관 비대칭: 한쪽 끝만 측정 → p₃ + p₄ 4점 옵션으로 완화

  - 원근 깊이 차이: 꼭대기와 밑동이 같은 평면에 없으면 d 단일 사용 부정확

[환경 한계]
  - QR이 나무에 가려지면 Phase 2 추적 불가 → 마지막 안정값 사용
  - 역광·우천 시 QR 인식률 저하
  - 강한 자기장 근처 → alpha(나침반) 부정확, beta/gamma는 가속도계·자이로로 산출하므로 영향 적음
  - iOS DeviceOrientation 권한 필요 (tilt.html에서 기구현)
```

```
  - 거리: 레이저 거리 측정기 (정확도 ±1cm)
  - 나무 높이: 줄자 또는 검증된 클리노미터 (정확도 ±5cm)

측정 대상: 반경 1m 이상 줄기 가진 수목, 주변 시야 확보
측정 조건:
  - 조도 조건 3종: 맑은 날 직사광선 / 흐린 날 / 그늘
  - 거리 3종: 5m / 8m / 12m
  - 기울기 범위: θ = 5°~35° (자연스럽게 발생)
```

#### 측정 프로토콜

```
1회 측정 세트:
  - 동일 나무에서 3회 반복 측정
  - 매 회 QR 재인식(Phase 1)부터 재시작
  - 결과: 높이 3회, 폭 3회

평가 지표:
  - 평균 오차: |측정값 − 참조값| / 참조값 × 100
  - RMSE: √(Σ(측정−참조)² / n)
  - 95% 신뢰구간
```

#### 합격 기준

| 항목 | 목표 | 최소 합격 |
|------|------|----------|
| 높이 평균 오차 | < 5% | < 10% |
| 폭 평균 오차 | < 8% | < 15% |
| 높이 RMSE | < 20cm | < 40cm |
| 반복 측정 표준편차 | < 5cm | < 10cm |
